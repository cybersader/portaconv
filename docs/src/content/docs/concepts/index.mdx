## The `Conversation` shape

<div style="background: var(--sl-color-gray-6); border-radius: 10px; padding: 1rem; margin: 1.5rem 0; font-family: var(--sl-font-mono); font-size: 0.85rem;">
<pre style="margin: 0;"><code>{`Conversation {
  id,                // session UUID
  title?,            // derived from first user message
  cwd?,              // launch-time working dir
  started_at?,
  messages: [
    Message {
      role,          // user | assistant | system | tool
      content: [
        Text { text } |
        ToolUse { id, name, input } |
        ToolResult { tool_use_id, output, is_error } |
        Thinking { text } |
        Unknown { raw }        ← resilience hook
      ],
      timestamp?,
      extensions,    // adapter-specific per-message bits
    }
  ],
  extensions,        // adapter-specific per-conversation bits
                     //   - system_events (dropped from messages)
                     //   - unknown_records (safety net)
}`}</code></pre>
</div>

<Aside type="note" title="Why extensions bags">
Every tool we'll ever adapt has fields the core schema doesn't promote (Claude Code has <code>gitBranch</code>, <code>permissionMode</code>, <code>thinkingMetadata</code>, <code>isCompactSummary</code>, …). Flattening them into the core pollutes the model; dropping them loses data. A <code>serde_json::Value</code> extensions bag at both levels is the third option — tool-specific bits survive the round trip without the core types having to know about them.
</Aside>

## Subagent filter

Subagent sessions are transient — they're called by a parent session's
tool_use, do their work, and return a consolidated result. Listing
them alongside top-level sessions in <code>pconv list</code> is noise
for paste workflows, so the adapter filters them out. Two shapes exist:

## Multi-session JSONLs

A single on-disk JSONL can carry multiple sessionIds when Claude's
<code>/compact</code> writes a continuation under a new session UUID
but appends to the same file. portaconv surfaces each distinct
sessionId as its own <code>SessionMeta</code>, matching Claude's
own <code>/resume</code> mental model where the sessionId — not the
file — is the identity.

<Aside type="caution" title="Tool-call state doesn't survive paste">
The renderer preserves the tool-call intent (name + args) and the result body, but the paste lands in a fresh agent process with no running tools and no live file state. Re-runs happen against the current filesystem. For most recovery scenarios this is fine — the substantive reasoning is what carries over.
</Aside>

## Read-only, on-demand

portaconv never writes to any tool's storage. There is no daemon,
no file watcher, no auto-sync. Every <code>pconv</code> invocation
reads the current state of disk and emits output. Simpler to reason
about, trivially reversible, composes with pipes.
